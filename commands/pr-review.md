# /pr-review - Automated PR Code Review

**Usage:** `/pr-review <pr-number>`

Reviews a GitHub pull request using Meridian's spec-reviewer agents, posting structured findings as a PR comment.

---

## Workflow

### Step 1: Fetch PR Data

Run both commands in parallel:

```bash
gh pr diff <number>
gh pr view <number> --json title,body,baseRefName,headRefName
```

Store the diff and PR metadata (title, body/description).

### Step 2: Prepare Review Context

- **plan**: Use the PR description (`body`) as the specification/plan document. If the body is empty, use the PR title as a one-line plan.
- **diff**: The raw diff from `gh pr diff`
- **changed_files**: Extract changed file paths from the diff headers (`--- a/` / `+++ b/` lines)

### Step 3: Run Spec Reviewers in Parallel

Launch **both** agents simultaneously via the Task tool:

#### Agent A — Compliance Review
```
subagent_type: "sf:spec-reviewer-compliance"
input:
  plan_file: "<inline: PR description as plan>"
  changed_files: [<list from diff>]
  diff_content: "<full diff>"
  review_mode: "pr"
```

#### Agent B — Quality Review
```
subagent_type: "sf:spec-reviewer-quality"
input:
  plan_file: "<inline: PR description as plan>"
  changed_files: [<list from diff>]
  diff_content: "<full diff>"
  review_mode: "pr"
```

Wait for both to complete before proceeding.

### Step 4: Collect and Categorise Findings

Merge findings from both agents into three buckets:

| Bucket | Criteria |
|--------|----------|
| **Must Fix** | Correctness errors, security issues, spec violations, broken tests |
| **Should Fix** | Code quality, performance concerns, missing error handling |
| **Suggestions** | Style, naming, optional improvements, documentation gaps |

### Step 5: Format Review Body

```markdown
## Meridian Code Review

> PR #<number>: <title>
> Reviewed by: spec-reviewer-compliance + spec-reviewer-quality

### Summary
<one paragraph: what was reviewed, overall impression>

### Must Fix
- [ ] `path/to/file.py:42` — Description of issue

### Should Fix
- [ ] `path/to/file.py:88` — Description of issue

### Suggestions
- `path/to/file.py:15` — Description of suggestion

---
*Auto-generated by Meridian /pr-review*
```

If a bucket is empty, write `_None found._`

### Step 6: Post Review Comment

```bash
gh pr review <number> --comment --body "<formatted review body>"
```

### Step 7: Report to User

Tell the user: PR number/title reviewed, count of Must Fix / Should Fix / Suggestions.

---

## Error Handling

| Error | Action |
|-------|--------|
| PR not found | Tell user: "PR #<number> not found. Check the number and that `gh` is authenticated." |
| Empty diff | Tell user: "PR #<number> has no diff (already merged or empty PR)." |
| Agent failure | Continue with the other agent's findings, note partial review |
| `gh` not installed | Tell user to install GitHub CLI: `brew install gh` |
